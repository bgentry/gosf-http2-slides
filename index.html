My goal is not to spend a lot of time talking about the basics of HTTP/2. I'm going to just briefly talk about the basics, but I encourage you to go learn more if you're not familiar with it.

HTTP/2

A quick intro

Why HTTP/2?

Performance.

  • End-user perceived latency
  • Network and server resource usage
  • More effective use of bandwidth

From an application's perspective, HTTP/2 works the same as HTTP/1.1. They are semantically identical.

HTTP/2 is just like HTTP/1.1*

*Semantically, at least.

The most important thing to remember about HTTP/2 is that it's still just HTTP. Same request methods, same status codes. URLs are not allowed to change. The same mental model of HTTP behavior is still valid.

Stuff that doesnt change

  • Request methods
  • Status codes
  • URLs
  • Other HTTP paradigms

Instead of a plaintext, loosely defined wire protocol, HTTP/2 uses a binary framing layer to encode HTTP messages on the wire.

They're still the same HTTP messages, they're just encoded differently. One of the best things about this is the format is very strictly defined, and as a result, it's much easier to write interoperable parsers for it.

HTTP/2

==

a new binary framing layer for HTTP

The other key differences that you've probably heard about:

  • Requests and responses are multiplexed. You can make many requests on the same connection, and receive many responses in any order. There's no head-of-line-blocking problem that made HTTP/1.1 pipelining so impractical.
  • Header compression: HTTP headers are extremely redundant and verbose, introducing a high overhead for each request (especially small ones). With HTTP/2's header compression scheme HPACK, the headers for a single request can be compressed into just a few bytes. This is especially helpful for small requests because many requests can now be compressed into a single packet.
  • TLS is not technically required by the spec but in practice will be. There was some drama around this. Chrome & Firefox leads have both said their implementations will be TLS-only.
  • HTTP/2 mandates TLS 1.2 or higher with strict security settings. Secure cipher suites, ephemeral keys, no renegotiation.
  • Less optional parts means less compatibility issues.

HTTP/2: Main Improvements

  • Multiplexed streams (requests & responses)
  • Header compression (HPACK)
  • Secure TLS required-ish
  • Less optional parts

Does it matter?

Admittedly, this is comparing a pathological, unoptimized case for HTTP/1.1.

We've evolved work-arounds to deal w/ this sort of problem, like domain sharding, image sprites, and other asset concatenation.

These hacks are actually harmful to HTTP/2 performance. Domain sharding increases latency variance, client & server load, and is more painful on high-latency or lossy mobile networks. Asset concatenation decreases cacheability and often means redownloading entire resources they're mostly unchanged. Spriting also means downloading assets that may not be needed on a given page.

Because HTTP/1.1 and HTTP/2 are going to coexist for a while, application developers will need to decide which target they want to optimize for, or develop techniques to optimize for both. You can imagine having a web server that's smart enough to toggle HTTP/1.1 hacks depending on the client.

HTTP/1.1 optimzations that are harmful to HTTP/2 perf:

  • Domain sharding
  • Image sprites
  • Inlining
  • Asset concatenation

Global State

Set data-state="something" on a slide and "something" will be added as a class to the document element when the slide is open. This lets you apply broader style changes, like switching the page background.

State Events
